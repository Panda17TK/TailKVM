# 復旧ルート（Recovery Routes）

入力転送系の不具合（カーソル拘束・キー押しっぱなし・操作不能）が起きたとき、
**ビルドし直しや再起動なしで即座に復旧する**ための経路一覧。上から順に試す。

## 1. Ctrl+Alt+Pause（物理フェイルセーフ・両側で有効）

- **コントローラ側**: 全キャプチャ（シームレス/フック/レガシー）を停止し、
  フックをアンインストール、カーソルクリップを解放、押下中のキー/ボタンを
  リモートへ解放送信する。
- **受信側（操作されているPC）**: 被制御セッションを即切断し、コントローラが
  押しっぱなしにしたキー/ボタンをすべて解放する（300ms間隔でポーリング）。

キーボードフックが死んでいても、シームレスループと受信ループが独立に
`GetAsyncKeyState` でポーリングしているため、どちらかは必ず効く。

## 2. トレイメニュー「Emergency reset (release all input)」

マウスが効かなくてもキーボード（Win キー → タブ移動）やタッチで到達できる、
ワンクリックの全解除:

1. 全転送停止（`pause_all_capture`: キャプチャフラグ・マウス/キーボードフック）
2. カーソルクリップ（ClipCursor）を無条件解放
3. Raw Input 診断キャプチャ停止
4. 被制御セッションの強制中断（`receiver_abort` → 受信ループが切断・スタック解放）

同じものが Tauri コマンド `emergency_reset` として UI からも呼べる。

## 3. 自動復旧（操作不要）

| ウォッチドッグ | 条件 | 動作 |
|---|---|---|
| リンク監視（コントローラ側） | リモート操作中に TCP セッション消失 1.5 秒 | 自動でローカル操作に復帰（クリップ解放・フック停止）。キャプチャはアーム維持で、再接続後はそのままエッジ越境が再び使える |
| ハートビート監視（受信側） | Heartbeat（2秒間隔）が 8 秒途絶 | セッション切断 + スタックキー/ボタン解放。コントローラのプロセスがクラッシュして FIN が来ないケースを救う |
| 受信応答監視（コントローラ側） | 相手からの受信（HeartbeatAck 等）が 8 秒途絶 | セッションを切断し supervisor がバックオフ付きで再接続。受信側の電源断・ネットワーク消失など TCP エラーにならないケースを救う。シームレスは sender スロット経由で再接続後に自動再開 |
| newest-wins（受信側） | 新しいコントローラが接続 | 旧セッションを置換。置換される側も退出時にスタック解放を実行 |
| 切断時解放（両側） | セッション終了（理由問わず） | 押下中として追跡していた全キー/ボタンを解放 |

## 4. 最終手段

1. トレイ「Quit」→ アプリ再起動（プロセス終了で OS が ClipCursor とフックを自動解放）
2. それも無理なら Ctrl+Alt+Del（セキュアデスクトップ遷移でフック/クリップが無効化される）

## 設計メモ

- 復旧経路は**互いに独立**にしてある（フック経由・ポーリング経由・TCP経由・トレイ経由）。
  単一の壊れたコンポーネントが全経路を道連れにしない。
- シームレスエンジンの送信は `controller_tx` スロットを送信時に解決するため、
  TCP 再接続後に自動で転送が復活する（再アーム不要）。
- 緊急リセットの `receiver_abort` はセッション開始時にクリアされるため、
  誰も操作していないときに押しても次のセッションを誤殺しない。
